Proposed Architecture

Tiered Multi-Agent Graveyard Shift (TAGS)

A multi-vendor orchestration model splitting high-context reasoning planners from low-cost concurrent execution workers, bound by credit pacing and quality feedback loops.
🧠 Tier 1: The Planners / Orchestrators
Owns the backlog triage, strategic sequencing, epic decomposition, and detailed worker speccing. Runs in a high-reasoning context.
Codex (Extra High Thinking) Gemini Pro Claude Opus (High)
⚙️ Tier 2: The Workers (Execution)
Executes targeted tasks concurrently using isolated git worktrees. Runs coding iteration, test diagnostics, and build updates.
Claude Sonnet / Opus (Medium) Codex (Medium) Gemini Pro (Equivalent to Sonnet)
⚖️ Tier 3: Quality Gates & Utilities
Runs automated test loops (tsc, cargo check, review loops) and uses lightweight utility parsing. Gemini Flash is relegated strictly to text parsing.
Static Verification Checks Codex PR Skill Gemini Flash (Parsing/Utility Only)

🛠️ Implementation Details
How to split the Orchestrator from the Worker pools
We can modify the graveyard-shift orchestrator code (running in the top-level model context) to dynamically spawn subagents using different models. When invoking invoke_subagent, the orchestrator specifies the targeted model inside the prompt or configurations:

"TypeName": "self" is used, but we parameterize the subagent's execution environment. For instance, the prompt given to the subagent instructs it to configure the runner to use the Flash model, while reserving the Pro model for planning.
Enforcing Concurrency & Worktree Safety
To prevent parallel workers from clashing on modifications, the orchestrator allocates a unique git worktree per issue. Each subagent executes within its own directory space (e.g., /Users/ray/Documents/skills-manager/.worktrees/issue-123). Once the worker completes, the worktree is automatically cleaned up or unlinked after the PR is successfully opened.
Escalation & Feedback Loops
If a worker model fails to compile or resolve tests within 3 iterations, it halts and reports its modifications and build logs back to the orchestrator. The orchestrator (running the high-reasoning model) analyzes the diff, determines if the error is architectural, and either:
  1. Corrects the spec and restarts the worker.
  2. Upgrades the task to a high-reasoning worker (e.g. Gemini Pro).
  3. Converts the task to a **Proposal** (drafting a 4-bullet comment or an epic design).
